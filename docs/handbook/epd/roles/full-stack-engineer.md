# Full-stack engineer

## TL;DR

- Full-stack engineers build the platform — ContentOS (Atlas) and CheckThat.ai.
- Rails + React stack. You work across the full stack, from database to UI.
- Part of the product lane, working in 4-week Shape Up cycles.
- Ship complete features, not just backend endpoints or frontend components.

## What this role does

Full-stack engineers build the core platform that powers GrowthX's products. You work across the entire stack — Rails backend, React frontend, database design, API architecture, and infrastructure. The products you build are the foundation that delivery teams and clients depend on every day.

You're part of the [product lane](/epd/product-lane), working alongside design engineers to ship features within Shape Up cycles.

## Key responsibilities

- **Feature development** — Build complete features end-to-end. From database schema to API endpoint to UI integration. Ship things that work.
- **Platform architecture** — Design systems that scale. Data models, API design, service boundaries, performance optimization.
- **Code quality** — Write clean, tested, maintainable code. Participate in code reviews. Maintain engineering standards.
- **Infrastructure** — Manage deployment pipelines, monitoring, and production infrastructure as needed.
- **Cross-stack problem solving** — Debug issues that span the stack. Performance bottlenecks, data inconsistencies, integration problems.

## Accountabilities

| What you own | How it's measured |
|-------------|-------------------|
| Feature delivery | Features shipped within cycle scope |
| Code quality | Test coverage, review feedback, production stability |
| Platform reliability | Uptime, performance, error rates |
| Technical debt | Balance between shipping fast and maintaining a clean codebase |

## What great looks like

- Features ship within Shape Up cycles — no carry-over work that bleeds into the next cycle
- Production is stable — things you build don't break
- You make pragmatic tradeoffs between speed and quality based on the bet's appetite
- Other engineers can understand and build on your code
- You understand the product context — why a feature matters to users, not just the technical spec

## How Shape Up works for you

Each 4-week cycle, the shaping team (Daniel, Renn, Marcel) defines the bets. You and the execution team have autonomy to solve the problem within the shaped boundaries.

**What this means in practice:**
- You won't get detailed technical specs. You'll get a pitch that explains the problem, the solution direction, and the appetite (time budget).
- You decide the architecture and implementation approach.
- You work in parallel with design engineers — they handle the experience, you handle the platform.
- Cooldown periods (2 weeks) are for tech debt, exploration, and personal projects.

See the [product dev process](/epd/product-dev-process) for the full methodology.

## Tools you'll use

- **Rails** — Backend framework
- **React** — Frontend framework
- **PostgreSQL** — Database
- **GitHub** — Code management, PRs, collaboration
- **Linear** — Task tracking within cycles
- **Cursor / AI tools** — AI-assisted development

## Key relationships

| Who | How you work together |
|-----|----------------------|
| Design engineers | Work in parallel during cycles. They own the experience; you own the platform. Coordinate on API contracts and data models. |
| Shaping team | They define the bets. You solve them. Ask questions early if something is unclear. |
| Other full-stack engineers | Code reviews, architecture discussions, shared patterns. |
| Client ops engineers | They build on the platform you maintain. Their needs inform platform decisions. |
| QA | Validate features work correctly before shipping. |

## Career path

- **Senior full-stack engineer** — Take on more complex architectural challenges, lead technical decisions for major features
- **Staff engineer** — Set technical direction across the platform, drive architectural initiatives
- **Engineering management** — Transition to leading the team if you want to focus on people and process
- **AI infrastructure** — Move to the AI infra lane if you want to work on the Output framework

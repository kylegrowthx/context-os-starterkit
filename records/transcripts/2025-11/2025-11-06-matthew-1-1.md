## Key Topics

### Workflow Bottlenecks & Tooling Issues

  - Usman's workload for clients Smith and Exec is unsustainable, leading to burnout.
  - **Tooling Conflict:**
      - **Claude:** Fast for drafting, but output is repetitive and requires heavy manual editing.
      - **Atlas:** Higher quality output, but the initial learning curve led to low trust and underuse.
  - **Smith Pipeline Issues:**
      - The current GenSeq pipeline splits artifacts (context, checklist) across multiple locations, making it hard to manage and causing missed instructions.
      - This forces Usman to use a simpler, but unsustainable, Claude-based process.

### Agentic Pipeline Philosophy

  - The core issue is using outdated, non-agentic pipelines.
  - **Agentic pipelines are superior because they are self-correcting:**
      - They use a recursive loop: Draft → Evaluate → Improve.
      - The agent evaluates its own output against instructions and automatically fixes errors.
  - **Key Principle:** The agent is a tool for *automation*, not a replacement for human judgment. The goal is to reduce manual work to a final 15-minute polish per article.

### Solution 1: Dedicated Comparison Article Pipeline

  - **Problem:** The Exec client requires \~40 comparison articles, each needing a specific table format that Usman currently adds manually.
  - **Solution:** Create a dedicated agentic pipeline for "Comparison Articles."
      - **Why:** This embeds the comparison table template as a default in the pipeline's code, eliminating repetitive manual setup.
      - **How:** Use the Biologica pipeline's simple, effective assignment directions as a model.
      - **Support:** Nana will assist with pipeline creation and template formatting.

### Solution 2: Refine the Proofreader Checklist

  - **Problem:** The Exec pipeline's "Proofreader Checklist" artifact is ineffective, forcing Usman to manually fix issues like m-dashes and long hyperlinks.
  - **Solution:** Refine the checklist to automate these fixes.
      - **Why:** This post-processing step runs after the initial draft, catching and correcting errors the main drafter missed.
      - **How:**
        1.  **Record Manual Fixes:** Document all common edits currently done in Claude.
        2.  **Synthesize Instructions:** Convert these into clear, specific commands for the checklist artifact.
        3.  **Test in Claude:** Validate the commands in a Claude project before implementing them in Atlas.
      - **Example:** The Biologica checklist's m-dash rule is highly effective: "Never use m-dashes. Instead, use commas when adding supplementary information. Do not use parenthetical constructions."
